## Goal
Track code churn (lines that were both added and then deleted/rewritten) per day as a metric for rework vs. net progress.

## Acceptance Criteria
- [ ] `daily_loc` table has a `churn` column
- [ ] Churn is calculated as `min(additions, deletions)` per day (a simple but effective approximation)
- [ ] `report_generator.py` displays churn in daily report and summary averages
- [ ] Churn ratio (`churn / (additions + deletions)`) shown in summary to indicate rework percentage

## Notes
Code churn = lines changed that cancel each other out. A simple approximation is `min(additions, deletions)` — if you add 100 and delete 80, ~80 lines were "churn" (modified/rewritten) and 20 were net new. More sophisticated approaches exist (per-file tracking, blame analysis) but this is a good starting signal.

## Implementation Plan
- [ ] Add `churn INTEGER NOT NULL DEFAULT 0` column to schema and migration in `init_db`
- [ ] Calculate churn as `min(total_add, total_del)` in the daily loop in `main()`
- [ ] Store churn in the INSERT statement
- [ ] Update `report_generator.py` daily report to show churn
- [ ] Add churn ratio to summary averages section
- [ ] Run `make clean && make` to verify
